## 解題說明

本題要求實作五種核心排序演算法，並針對不同的資料量 $n$（$500, 1000, 2000, 3000, 4000, 5000$）在最壞情況（Worst-case）與平均情況（Average-case）下的執行時間與記憶體用量進行精確的動態計時與實測分析。

要實作與評估的演算法包含：
1. **Insertion Sort（插入排序）**
2. **Quick Sort（快速排序）**：規定必須使用 **Median-of-three（三數取中法）** 來選擇 Pivot。
3. **Merge Sort（合併排序）**：規定必須使用 **Iterative（迭代法/非遞迴）** 版本實作。
4. **Heap Sort（堆積排序）**。
5. **Composite Sort（複合排序法）**：結合多種排序法（如大資料量使用 Quick Sort，小資料量或幾乎排好時切換至 Insertion Sort），透過實驗找出臨界點（Threshold），使其在各種 $n$ 下均能達到最優化速度。

## 解題策略與魔鬼細節

### 1. 測試資料建立與保留機制
為了確保實驗的公平性與精確度，測資生成遵循以下嚴格標準：
* **Average-case**：使用隨機數生成測資。同一組隨機產生的原始數組必須**同時複製**給五種排序法使用，避免因資料隨機性分佈不均產生誤差。在傳遞數組至排序函式時使用「傳值（Pass by Value）」而非「傳參考（Pass by Reference）」，防止前一個排序法將資料排好。
* **Insertion Sort (Worst-case)**：直接生成逆序數組 $[n, n-1, n-2, \dots, 1]$。
* **Merge Sort & Heap Sort (Worst-case)**：這兩種演算法難以從數學上直接逆推最壞測資。策略上採用助教提示的 **`Permute` 函式隨機打亂**。針對每個 $n$，連續生成至少 10 組以上的隨機排列進行盲測，**記錄並保留使該演算法執行時間最長的極端數組**作為 Worst-case 測試資料。
* **Quick Sort (Worst-case)**：配合 Median-of-three 策略，透過反轉排序邏輯或使用隨機大樣本（10組以上排列）抓取最大執行時間之測資，使其切分區間達到最不均勻狀態。

### 2. 高精度計時與重複實驗
由於現代 CPU 效能極高，在 $n \le 5000$ 的量級下，單次排序時間可能趨近於 0 微秒，落入時鐘精度誤差範圍（$\delta$）。
* **解決策略**：在外層使用 `for (int i = 0; i < 1000; ++i)` 進行**重複執行循環**，測量累積總時間後再除以重複次數，以獲得準確度在 1% 以內的微秒（$\mu s$）級別數據。

### 3. 記憶體使用量增量測量
使用 Windows API `<Psapi.h>` 的 `GetProcessMemoryInfo` 追蹤結構體中的 `WorkingSetSize`（實體記憶體分頁量）。為了排除作業系統環境雜訊，不直接輸出單次 KB 值，而是透過函式執行前、中、後的**記憶體量增量計算**，反推其動態配置的空間複雜度。

## 申論及開發報告

時間效能趨勢：當資料量 $n$ 從 500 成長至 5000 時，可以明顯觀察到 Insertion Sort 的時間呈現二次方（Quadratic）的暴增趨勢。相較之下，Quick, Merge, Heap Sort 則呈現穩定的高效率線性對數（$n \log n$）成長。Composite Sort 在隨機分布與大資料量時，其耗時皆能比純高階排序法低。這印證了「在子陣列極小時，利用 $O(n^2)$ 的 Insertion Sort 代替遞迴分支，能有效減少運算常數與 Cache miss」的理論基礎。空間增量與理論轉換：在精準定位的記憶體測量中，唯有 Merge Sort 在配置 temp 陣列的期間出現了顯著的 KB 級增量，且該增量隨著資料量 $n$ 的增加呈線性比例（Linear）成長，這完美契合了其額外空間複雜度為 $O(n)$ 的理論預測。而其餘排序法在執行期間的記憶體增量皆保持為 0（或極低常數變數開銷），成功驗證了其 $O(1)$ 的空間複雜度。
